# Graphsmith Planner Semantics (v1)

## What the planner receives

| Field | Type | Required | Description |
|-------|------|----------|-------------|
| `goal` | `str` | yes | Natural-language task description |
| `candidates` | `list[IndexEntry]` | yes | Skills retrieved from the registry |
| `constraints` | `list[str]` | no | Additional restrictions (e.g. "no network effects") |
| `desired_outputs` | `list[IOField]` | no | Expected output fields and types |
| `max_candidates` | `int` | no | Cap on candidate skills (default 20) |

## What the planner returns

A `PlanResult` — one of three outcomes:

1. **Success** — a valid `GlueGraph` with no unresolved holes.
2. **Partial** — a `GlueGraph` plus a list of `UnresolvedHole` entries.
3. **Failure** — no graph produced; a list of reasons.

```
PlanResult
  status: "success" | "partial" | "failure"
  graph: GlueGraph | None
  holes: list[UnresolvedHole]
  reasoning: str          # human-readable planner reasoning
```

All three cases are typed structured objects.
The graph is never raw YAML/JSON from the LLM — it is always a
constructed `GlueGraph` model instance.

## What is a glue graph

A **glue graph** is a task-specific composed graph that:

- has a `goal` string (the user request that produced it)
- has typed `inputs` and `outputs` (like a skill)
- has a `GraphBody` (nodes, edges, outputs)
- may reference published skills via `skill.invoke` nodes
- may use any primitive op
- is **not** automatically publishable as a reusable skill
- can be validated by the existing validator (via a synthetic SkillPackage)

The distinction from a SkillPackage:
- A SkillPackage is a reusable, published unit.
- A GlueGraph is a task-specific artifact generated by the planner.

## Unresolved holes

An `UnresolvedHole` has:

| Field | Type | Description |
|-------|------|-------------|
| `node_id` | `str` | Which planned node has the gap |
| `kind` | `str` | Category: `missing_skill`, `ambiguous_candidate`, `missing_output_path`, `unsupported_op` |
| `description` | `str` | Human-readable explanation |
| `candidates` | `list[str]` | Skill IDs considered (if any) |

Holes are structured data, not free text.
They appear in CLI output and are testable.

## Candidate skill retrieval

1. Split the `goal` into keyword tokens.
2. For each token, run `registry.search(token)`.
3. Deduplicate by `(id, version)`.
4. Sort by `(id, version)` for determinism.
5. Truncate to `max_candidates`.

No embeddings. No reranking. Deterministic.

## Planner backend interface

```python
class PlannerBackend(Protocol):
    def compose(self, request: PlanRequest) -> PlanResult: ...
```

Implementations:
- `MockPlannerBackend` — returns canned plans from fixtures for testing.
- Future: `LLMPlannerBackend` — calls an LLM provider (not in v1 scope).

## Validation of planner output

Before a `PlanResult` is returned to the user:

1. If a `GlueGraph` is present, wrap it in a synthetic `SkillPackage`.
2. Run `validate_skill_package()` on it.
3. If validation fails, demote to `partial` or `failure` status
   and attach the validation error as a hole.

This reuses the existing validator — no separate validation layer.

## What is NOT allowed

- Planner-generated arbitrary Python code.
- Automatic execution of the planned graph.
- Automatic publication of glue graphs.
- Unbounded loops in generated graphs.
- Ops outside the PRIMITIVE_OPS vocabulary.
