# Architecture

Graphsmith separates semantic reasoning (LLM) from structural correctness
(deterministic compiler). This page explains each stage of the pipeline.

## Pipeline overview

```
User goal (natural language)
    │
    ▼
1. Retrieve candidate skills from registry
    │
    ▼
2. Semantic decomposition (LLM)
    → content_transforms: ["normalize", "extract_keywords"]
    → presentation: "none" | "list" | "header"
    → final_output_names: ["normalized", "keywords"]
    │
    ▼
3. Generate N IR candidates (LLM × N)
    → Each: steps, sources, config, final_outputs
    │
    ▼
4. Score + rerank (deterministic)
    → Penalize unnecessary formatting
    → Reward required skills
    → Enforce decomposition consistency
    │
    ▼
5. Compile best candidate (deterministic)
    → IR → GraphNodes, GraphEdges, graph_outputs
    │
    ▼
6. Validate (deterministic)
    → Types, ops, edges, DAG, bindings, outputs
    │
    ▼
7. Execute (topological runtime)
```

## Components

### Semantic decomposition

**Module**: `graphsmith/planner/decomposition.py`

Before IR generation, an LLM analyzes the goal and produces a compact semantic
classification:

- **content_transforms** — what processing steps are needed (normalize, summarize, etc.)
- **presentation** — how to format the output (none, list, header)
- **final_output_names** — which skill output ports to expose

This makes semantic decisions explicit and constrains IR generation.

A deterministic fallback handles LLM parse failures using keyword matching.

### Planning IR

**Module**: `graphsmith/planner/ir.py`

The LLM emits a semantic intermediate representation (IR), not a raw graph:

```python
PlanningIR:
    goal: str
    inputs: [IRInput(name, type)]
    steps: [IRStep(name, skill_id, version, sources, config)]
    final_outputs: {name: IROutputRef(step, port)}
    effects: [str]
```

The IR is deliberately smaller than a graph — no edge addresses, no node IDs,
no graph_outputs mapping.

### Deterministic compiler

**Module**: `graphsmith/planner/compiler.py`

The compiler lowers IR to a fully-formed GlueGraph:

1. **Sanitize** step names (dots, spaces → underscores)
2. **Validate** IR consistency (references, cycles, self-loops)
3. **Build nodes** from steps (skill.invoke vs primitive op)
4. **Build edges** from sources (scope.port addressing)
5. **Build outputs** from final_outputs
6. **Normalize** types and skill_ids

All decisions are deterministic. Structured `CompilerError` subclasses carry
phase and details for diagnostics.

### Semantic scorer

**Module**: `graphsmith/planner/ir_scorer.py`

Ranks candidates without LLM involvement:

| Signal | Points |
|--------|--------|
| Unnecessary formatting step | -20 |
| Required skill present | +10 |
| Required skill missing | -15 |
| Wrong skill family | -15 |
| Correct output name | +5 |
| Output name mismatch | -10 |
| Decomposition consistency | ±8 to ±25 |

### Candidate reranking

**Module**: `graphsmith/planner/ir_backend.py`

With `candidate_count > 1`, the backend:

1. Generates N independent IR candidates
2. Parses + compiles + validates each
3. Scores each with the deterministic scorer
4. Selects the highest-scoring valid candidate

### Validator

**Module**: `graphsmith/validator/core.py`

10+ checks: effects, types, node IDs, ops, edges, binding conflicts,
required inputs, output declarations, DAG property.

### Runtime

**Module**: `graphsmith/runtime/executor.py`

Topological executor with flat value store. Deterministic except for LLM
provider output. Supports nested `skill.invoke`.

## Design principles

1. **LLM handles semantics, Graphsmith handles determinism** — the LLM chooses
   what skills to use; the compiler builds the graph structure.

2. **No raw graph editing by LLM** — the LLM never writes edge addresses or
   node IDs. These are generated by the compiler.

3. **Candidate diversity over single-shot correctness** — multiple candidates +
   deterministic scoring is more robust than one perfect answer.

4. **Errors are structural, not prompt hacks** — when something goes wrong,
   classified error types identify the phase and details.
