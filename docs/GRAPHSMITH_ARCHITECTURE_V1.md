# Graphsmith Architecture v1

## End-to-end flow

```
User goal (natural language)
    │
    ▼
Retrieve candidate skills from registry
    │
    ▼
Decompose goal (LLM)
  → content_transforms: ["normalize", "extract_keywords"]
  → presentation: "none" | "list" | "header"
  → final_output_names: ["normalized", "keywords"]
    │
    ▼
Generate N IR candidates (LLM × N), conditioned on decomposition
  → Each candidate: steps, sources, config, final_outputs
    │
    ▼
Score each candidate (deterministic semantic scorer)
  → Penalize: unnecessary formatting, missing skills, wrong families
  → Reward: required skills present, correct output names
  → Enforce: decomposition consistency
    │
    ▼
Compile best candidate (deterministic compiler)
  → IR steps → GraphNodes (node IDs, ops, config)
  → IR sources → GraphEdges (scope.port addresses)
  → IR final_outputs → graph_outputs mapping
  → Sanitize step names, normalize types
    │
    ▼
Validate compiled graph (deterministic validator)
  → Types, ops, edges, DAG, bindings, outputs
    │
    ▼
Execute (topological runtime)
```

## Components

### Planning IR (`planner/ir.py`)

Semantic intermediate representation. What the LLM emits:

```python
PlanningIR:
  goal: str
  inputs: [IRInput(name, type)]
  steps: [IRStep(name, skill_id, version, sources, config)]
  final_outputs: {name: IROutputRef(step, port)}
  effects: [str]
```

The IR is deliberately smaller than a graph. No edge addresses, no node IDs,
no `graph_outputs` mapping. The compiler generates all of those.

### Compiler (`planner/compiler.py`)

Deterministic IR → GlueGraph lowering:

1. **Sanitize** step names (dots, spaces → underscores, collision resolution)
2. **Validate** IR consistency (references, effects, cycles, self-loops)
3. **Build nodes** from steps (skill.invoke vs primitive op detection)
4. **Build edges** from sources (scope.port address construction)
5. **Build outputs** from final_outputs
6. **Normalize** input types (invalid types → string)
7. **Normalize** skill_ids (strip @version suffixes)

Structured error types (`CompilerError` subclasses) carry `phase` and `details`
for diagnostics.

### Decomposition (`planner/decomposition.py`)

Semantic pre-planning stage. Makes intent explicit before IR generation:

```python
SemanticDecomposition:
  content_transforms: ["normalize", "extract_keywords"]
  presentation: "none" | "list" | "header" | "template"
  final_output_names: ["keywords"]
```

The decomposition is injected into the IR prompt as a binding contract.
Deterministic fallback handles LLM parse failures.

### Deterministic scorer (`planner/ir_scorer.py`)

Ranks candidates without LLM involvement:

| Signal | Points |
|--------|--------|
| Unnecessary formatting when not requested | -20 |
| Required skill present | +10 |
| Required skill missing | -15 |
| Wrong skill family (JSON for text) | -15 |
| Output name matches skill port | +5 |
| Output name mismatch | -10 |
| Excess steps | -5 per extra |
| Decomposition consistency | ±8 to ±25 |

### Candidate reranking (`planner/ir_backend.py`)

`IRPlannerBackend` with `candidate_count > 1`:
1. Generate N IR candidates (independent LLM calls)
2. Parse + compile + validate each
3. Score each with deterministic scorer
4. Select highest-scoring valid candidate

### Validator (`validator/core.py`)

10+ deterministic checks: effects, types, node IDs, ops, edges, binding
conflicts, required inputs, output declarations, DAG property (Kahn's algorithm).

### Runtime (`runtime/executor.py`)

Topological executor with flat value store. Supports nested `skill.invoke`.
Optional inputs skipped gracefully. Deterministic except for LLM provider output.

## Design principles

1. **LLM handles semantics, Graphsmith handles determinism.**
   The LLM chooses what skills to use and how data flows. Graphsmith compiles
   the structural graph, validates it, and executes it.

2. **No raw graph editing by LLM.**
   The LLM never writes edge addresses, node IDs, or graph_outputs mappings.
   These are all generated by the compiler from the semantic IR.

3. **Errors are structural, not prompt hacks.**
   When something goes wrong, structured error types identify the phase and
   details. No "try again" loops — the compiler either succeeds or reports
   a classified error.

4. **Candidate diversity over single-shot correctness.**
   Multiple candidates + deterministic scoring is more robust than trying to
   get one perfect answer from the LLM.

## Why this works

- **Structural errors eliminated**: invalid edges, types, self-loops, cycles
  are impossible because the compiler generates all graph mechanics.
- **Semantic variance reduced**: decomposition makes intent explicit;
  reranking selects the best among diverse candidates.
- **Candidate diversity leveraged**: even if individual candidates have
  ~60% quality, picking from 3 candidates with good scoring reaches ~92-100%.

## Known limitations

- **Weak-model semantic noise**: Llama 3.1 8B has 8 intermittent goals (wrong
  output names) that vary between runs.
- **Skill registry scope**: 15 skills covering text + JSON. Broader skill
  families untested.
- **No learned components**: scorer is hand-written rules. Learned reranker
  prototype exists but has no headroom on Claude Haiku.
- **No repair loop**: if all N candidates are wrong the same way, the system
  cannot self-correct.
